# Independent Strategic Review: MKUltraman v4 Launch Plan

## TL;DR

This plan solves the wrong problem and optimizes for technical metrics over business outcomes. The focus on Lighthouse scores and loading speed ignores fundamental positioning confusion that's killing conversions. The 6-month regret scenario is shipping a faster version of the same fragmented identity.

## Findings

### 1. Problem Reframing Failure
**What's Wrong:** Plan treats site performance as the core issue when the audit clearly identifies positioning drift as the primary conversion killer.
**Severity:** Critical
**Fix:** Reframe around market positioning first. The current site confuses visitors with 4 different identities (strategist/SEO agency/consultant/commentator). No amount of speed fixes that.

### 2. Unstated Premises Risk
**What's Wrong:** Plan assumes SME owners in Japan prioritize site speed over trust signals. Zero validation of this premise.
**Severity:** High  
**Fix:** Test the hypothesis that performance drives inquiries. Japanese business culture heavily weights social proof and relationship building over technical specs.

### 3. Competitive Blindness
**What's Wrong:** No competitive analysis of other Japan-focused consultants. Plan assumes speed = differentiation when competitors may already have this solved.
**Severity:** High
**Fix:** Map the competitive landscape. If others have fast sites with clear positioning, this plan delivers table stakes, not advantage.

### 4. Trust Architecture Gap
**What's Wrong:** Plan adds proof strips and stats but ignores the audit's finding that "homepage jumps from claim to form without enough trust scaffolding."
**Severity:** Medium
**Fix:** Build case studies first, performance second. Japanese buyers need relationship proof over loading metrics.

### 5. Technical-First Execution Bias
**What's Wrong:** 75% of implementation time allocated to CSS tokens, redirects, and Lighthouse optimization. Only 25% on conversion-critical trust elements.
**Severity:** Medium
**Fix:** Invert the priorities. Ship positioning clarity and proof before technical polish.

## Implications

The plan optimizes for developer satisfaction (clean builds, good Lighthouse scores) over customer conversion. This is a classic technical founder trap - solving the measurable problem instead of the revenue problem.

The audit explicitly states the prototype "feels like a disciplined wireframe with strong copy instincts, not yet a fully persuasive flagship." The plan doesn't address this gap.

## Recommended Next Branches

1. **Positioning Lock Sprint:** Define single identity (operator vs strategist vs consultant) before touching code
2. **Trust System Design:** Create proof architecture (case studies, quantified outcomes) before homepage polish  
3. **Competitive Intelligence:** Map Japan consultant landscape to identify true differentiation vectors
4. **Conversion Funnel Mapping:** Track visitor flow from awareness to inquiry submission

## Open Questions

- How many current site visitors actually abandon due to loading speed vs unclear value prop?
- What do successful Japan-focused consultants' sites prioritize?
- Is the SME owner persona validated or assumed?
- What's the actual conversion rate problem - traffic volume or visitor quality?

The plan executes well on technical delivery but misses the strategic foundation that would make technical excellence meaningful.
